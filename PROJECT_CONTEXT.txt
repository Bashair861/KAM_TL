WHAT IS THIS PROJECT?
────────────────────────────────────────────────────────────────────────────────
Aether KAM is an internal Key Account Management (KAM) portal for an IT services
company. It is used by the KAM team to manage enterprise client relationships,
track account health, manage escalations, handle contracts, and grow accounts.

The app is built as a learning/portfolio project. It started as a fully static
mock-data prototype (all data was hardcoded in a TypeScript file). We are
progressively wiring it to a real backend (Supabase) and making it production-
ready.

USER ROLES & PERMISSIONS (ROLE_PERMISSIONS in kam-data.ts)
────────────────────────────────────────────────────────────────────────────────
Three roles exist:

  CEO          → read: true,  write: false, scope: "all"
  Head of KAM  → read: true,  write: true,  scope: "all"
  KAM          → read: true,  write: true,  scope: "own"

- "scope: all"  = can see/edit all accounts
- "scope: own"  = can only edit accounts assigned to them
- Write access controls whether Save/Edit/Add buttons are enabled in the UI.
- Role is stored in the `profiles` table in Supabase.
- Role is loaded into React via AuthContext → useAuth() → profile.role

IMPORTANT DECISIONS & PATTERNS
────────────────────────────────────────────────────────────────────────────────

1. DATA LAYER PATTERN (db.ts)
   - All Supabase calls live in src/services/db.ts
   - Mapper functions convert snake_case DB columns → camelCase TypeScript types
   - fetchAccount() uses Promise.all for 7 parallel queries (performance)
   - Components never import from @supabase/supabase-js directly

2. AUTH PATTERN
   - AuthContext is the single source of truth for auth state
   - session = Supabase Session object (null if logged out)
   - profile = row from profiles table (loaded by user UUID or email fallback)
   - loading = true until first session check completes (prevents flash of wrong UI)
   - Route protection is centralized in AppShell in __root.tsx, NOT in individual routes

3. PERMISSION PATTERN
   - Every component that needs edit-gating calls: const { profile } = useAuth()
   - Then: const editable = ROLE_PERMISSIONS[profile?.role ?? "KAM"].write
   - Buttons use disabled={!editable}
   - The "KAM" fallback means unknown roles default to most restrictive write scope

4. ENVIRONMENT VARIABLES
   - VITE_SUPABASE_URL     → browser-safe, used by src/lib/supabase.ts
   - VITE_SUPABASE_ANON_KEY → browser-safe publishable key (sb_publishable_...)
   - SUPABASE_SERVICE_ROLE_KEY → server/seed only, bypasses RLS (sb_secret_...)
     NEVER put the service role key in a VITE_ prefixed variable

5. SEEDED PROFILES vs REAL AUTH USERS
   - Profiles seeded with static UUIDs (00000000-...-0001, 00000000-...-0002)
   - Real Supabase Auth users have random UUIDs
   - loadProfile() handles this: tries UUID first, falls back to email lookup
   - Long-term fix: run setup-auth.sql trigger + UPDATE queries after creating users

6. DEBUG ERROR BANNER (dashboard)
   - Dashboard (index.tsx) has a visible error banner showing raw DB errors
   - This was intentionally kept (user request: "leave debug error as is for now")
   - Will be cleaned up in a future phase

7. RLS (Row Level Security)
   - RLS is ENABLED on all 13 tables
   - Public SELECT policies allow the anon key to read all data
   - Seed script uses service role key to bypass RLS during insert/upsert
   - Never disable RLS — Supabase shows critical security warnings
